두그루 (dogroo)

서비스 종료되는 그루우(groo)를 대신할, 화분 80개를 위한 개인용 물주기·분갈이 관리 PWA

배경 & 목표

  • 식물관리 앱 그루우(groo)가 서비스 종료 예정. 기존 데이터는 백업(CSV 예상)으로 제공됨.
  • 사용자 1명. 현재 화분 약 80개, 식물 종류는 계속 추가될 수 있음.
  • 필요한 것은 딱 두 가지 관리 축 — 물주기와 분갈이. 부가로 키우기 시작한 날짜, 사진, 메모.
  • 완전 개인 용도: 앱스토어 배포 없음, 서버는 집의 개인 x86 리눅스 머신 + iptime DDNS(개인 도메인 없이 DDNS 주소만 사용).
  • 지원 기기: iPhone 17 · iPhone 15 Pro.

UX 원칙

  • 한눈에, 심플하게 — 모바일 단일 컬럼. 홈에 들어오면 "오늘 뭘 해야 하는지"가 바로 보인다.
  • 원탭 완료 — 물주기·분갈이 체크는 목록에서 탭 한 번. 잘못 누르면 토스트로 즉시 실행취소.
  • 식물 사진이 주인공 — 목록·상세 모두 사진 중심, 장식 최소화.
  • 상태는 색으로 — 지남(빨강) / 오늘(주황) / 여유(초록) D-day 배지.
  • 큰 터치 타깃 — 매일 쓰는 버튼은 엄지 기준으로 크게.

핵심 기능

🪴 식물(화분) 관리

  • 등록·수정: 이름, 종(식물 풀에서 검색·선택), 키우기 시작일, 메모
  • 사진: 카메라 촬영 / 앨범 불러오기 / 교체 / 삭제, 여러 장(성장 기록)
  • 떠나보낸 화분은 삭제 대신 보관(아카이브)

💧 물주기 · 🏺 분갈이 체크

  • 목록·상세 어디서든 원탭 완료 기록 (오늘 날짜)
  • 완료 직후 5초간 실행취소 토스트 (잘못 누름 대비)
  • 소급 기록 — 기록을 깜빡한 경우 완료 버튼을 길게 눌러 어제 ~ 5일 전 중 선택. 다음 예정일도 그 날짜 기준으로 재계산
  • 마지막 기록 기준 다음 예정일 자동 계산, D-day 상태 배지

📋 목록 사용성

  • 정렬: 임박순(기본) / 이름순 / 최근 등록순
  • 필터: 오늘 할 일만 / 물주기·분갈이별 / 종별 / 보관 포함
  • 검색: 이름·종 즉시 검색
  • 당겨서 새로고침 (pull-to-refresh)

✨ 주기 추천

  • 서버에 저장된 종(species) 데이터 + 계절 기반으로 물주기·분갈이 주기 추천
  • 화분 크기(소/중/대)에 따라 물주기 추천 보정 — 작을수록 빨리 마르므로 주기 단축
  • 상세 화면에 "추천 5일 (여름·중형)" 표시, 원탭 [적용]
  • 추천 주기는 정렬(임박순)·알림 계산에도 반영

🔔 알림 & 설정

  • 알림 종류별 on/off — 물주기 예정, 분갈이 예정, 미완료 리마인더(저녁 재알림)
  • 발송 시각 조정 (아침 1회 기본), 전체 알림 일괄 off
  • 알림 문구에 대상 화분 수·이름 요약

🔐 로그인

  • 아이디 / 비밀번호 로그인 (외부 공개이므로 필수)
  • 비밀번호는 해시 저장, "로그인 유지" 체크
  • 로그인 후에만 모든 화면 접근 가능

비스코프: 커뮤니티·식물 도감·다중 사용자·병해충 진단 등 groo의 부가 기능은 만들지 않는다.

기술 결정

항목결정근거
플랫폼 PWA (홈화면 설치) iOS 16.4+에서 홈화면 설치 시 푸시·카메라 지원. 서명 만료 리스크 없고 유지보수가 가장 단순.
프론트엔드 Lit + TypeScript 사용자 선택. 가볍고 웹 표준(Web Components) 기반.
푸시 알림 Web Push (VAPID) — web-push npm Firebase 불필요. Apple 엔드포인트(web.push.apple.com)로 직접 발송. 완전 무료, 외부 서비스 의존 0. 단, 홈화면에 설치한 PWA에서만 수신 가능. 종류별 on/off·발송 시각은 앱 설정에서 제어.
백엔드 Caddy + Fastify (Node, TypeScript) Node 개발자 선호 스택. 이 규모에 충분. Caddy가 HTTPS 인증서 자동 관리.
DB SQLite (better-sqlite3) 사용자 1명, 일일 IO 소량. 운영 부담 0, 백업은 파일 복사 한 번, 서버 재설치·이전도 간단.
사진 저장 파일시스템, DB에는 경로만 확정 사항. 리사이즈된 썸네일 별도 생성(sharp) 검토.
서버 개인 x86 리눅스 (재설치 예정) + iptime DDNS 개인 도메인 없이 DDNS 주소만 사용. OS는 Ubuntu Server LTS 권장 (Node·Caddy 패키지 지원 무난). 포트포워딩 80/443.
인증 아이디/비밀번호 + users 테이블 + 세션 쿠키 외부(DDNS)에 공개되므로 로그인 필수. 비밀번호는 argon2id 해시로 저장. @fastify/session 세션 쿠키, "로그인 유지" 장기 세션 지원.

데이터 모델 초안

users                          -- 로그인 계정
  id            INTEGER PK
  username      TEXT UNIQUE
  password_hash TEXT           -- argon2id
  created_at    TEXT

species                        -- 식물 종 정보 (주기 추천의 기준)
  id            INTEGER PK
  name          TEXT UNIQUE    -- 식물 풀(species-pool.sql)로 초기 등록
  water_summer_days   INTEGER  -- 여름철 권장 물주기
  water_winter_days   INTEGER  -- 겨울철 권장 물주기
  repot_months        INTEGER  -- 권장 분갈이 주기
  memo          TEXT

plants                         -- 화분(개체) 단위
  id            INTEGER PK
  name          TEXT           -- 애칭
  species_id    INTEGER FK NULL
  started_at    TEXT           -- 키우기 시작일 (ISO date)
  pot_size      TEXT           -- 'S' | 'M' | 'L' (물주기 추천 보정)
  water_interval_days   INTEGER  -- 사용자 확정값 (추천은 참고)
  repot_interval_months INTEGER
  memo          TEXT
  archived_at   TEXT NULL      -- 보관(아카이브) 처리

photos
  id, plant_id FK, path, taken_at, is_primary

watering_logs
  id, plant_id FK, watered_at, memo

repotting_logs
  id, plant_id FK, repotted_at, pot_size, memo   -- 새 화분 크기도 기록 (성장 이력)

push_subscriptions
  id, endpoint, keys_p256dh, keys_auth, created_at

notification_settings          -- 알림 종류별 설정
  type          TEXT PK        -- 'watering' | 'repotting' | 'reminder'
  enabled       INTEGER        -- 0/1
  send_at       TEXT           -- 발송 시각 (HH:MM)

다음 물주기 예정일 = 마지막 watering_logs.watered_at + water_interval_days (별도 컬럼 없이 조회 시 계산). 분갈이도 동일 방식.

주기 추천 로직

  • 물주기 추천 = 종의 계절별 권장값 × 화분 크기 보정 — 현재 월로 계절 판정 (예: 4~9월 여름 주기, 10~3월 겨울 주기) 후, 화분 크기 보정 적용: 소형 ×0.8 (빨리 마름) / 중형 ×1.0 / 대형 ×1.3 (천천히 마름), 반올림.
  • 분갈이 추천은 크기 보정 없이 종의 권장 주기(repot_months)만 사용 — 분갈이 시점은 생장 속도가 지배적이라 시간 기준이 실용적. 대신 분갈이 기록에 새 화분 크기를 남겨 성장 이력·다음 추천 보정에 활용.
  • 재질(토분/플라스틱)·배수·채광 등 추가 변수는 입력 부담 대비 효과가 작아 비스코프 — 사용자 확정값 조정으로 흡수.
  • 추천은 어디까지나 제안 — 사용자가 [적용]해야 plants의 확정값이 바뀜. 알림·임박순 정렬은 확정값 기준.
  • 종 정보가 없는 식물은 추천 없이 사용자 입력값만 사용.

데이터 이전 & 백업

  • groo 백업 파일(CSV 예상) 임포트 스크립트 — 실물 스키마 확인 후 작성.
  • 백업 = SQLite 파일 + 사진 디렉토리 복사, cron으로 일일 자동화 (외부 저장소 복사).

로드맵

  1. 서버 셋업 — OS 재설치(Ubuntu), Caddy, iptime DDNS·포트포워딩, HTTPS 확인
  2. 백엔드 — Fastify + SQLite 스키마(users·species 포함), 식물/물주기/분갈이 CRUD API, 아이디/비밀번호 로그인(argon2id 해시)
  3. 프론트 — Lit PWA (manifest, Service Worker), 목록·상세·등록·설정 화면, 사진 업로드, 로고 에셋 용도별 리사이즈·manifest/홈화면 아이콘 세트 생성
  4. 푸시 알림 — VAPID 키 생성, 구독 등록, 알림 설정 화면(종류별 on/off·시각), 스케줄 발송
  5. 주기 추천 — species 계절별 권장 주기 데이터 구축, 추천 API·[적용] UI
  6. groo 데이터 이전 — 백업 CSV 스키마 확인 후 임포트 스크립트
  7. 백업 자동화 — DB 파일 + 사진 디렉토리 일일 백업 (외부 저장소 복사)

미결 사항 (Open Questions)

  • 서버 OS — Ubuntu Server LTS로 제안, 재설치 시점에 확정.
  • species 권장 주기 보완 — 백업 도감(wiki)에 물주기 간격이 44종 중 9종만 존재. 나머지는 사용하면서 수동 보완. (백업 데이터는 확보 완료 — 마이그레이션 전략은 docs/migration.md)
